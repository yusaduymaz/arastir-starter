# Phase 14: Real-Time UX & Polish
## Plan 14-01: True Parallel Real-Time Progress

### Objective
Fix the orchestrator pipeline so that the frontend accurately reflects the parallel execution of agents in real-time, eliminating the "jumps from 5% directly to 100%" bug.

### Context & Rationale
Currently, the orchestrator (`src/app/api/research/route.ts`) launches 4 data agents using `Promise.allSettled`. While the agents do run in parallel, the orchestrator waits for all of them to finish before it logs anything to the `research_sessions` table. Because the frontend (`ActivePipeline.tsx`) listens only to `research_sessions`, the user sees 5% for 15-20 seconds, and then all logs dump at once. 
To achieve true real-time, the frontend must subscribe to the `agent_runs` table, or the agents must emit progress back to `research_sessions` *while* they run. The most architecturally sound approach (intended in Phase 11) is for the frontend to subscribe to `agent_runs` for live agent states.

### Scope
- Update `ActivePipeline.tsx` to subscribe to the `agent_runs` table via Supabase Realtime, filtering by `session_id`.
- Update the progress bar to calculate its percentage based on the number of completed `agent_runs` rather than a hardcoded `progress` integer from the orchestrator.
- Ensure the orchestrator's `Promise.allSettled` block doesn't artificially gate the frontend from seeing running states.

### Implementation Steps
1. **Frontend Subscription**: In `ActivePipeline.tsx`, add a second `supabase.channel` subscription that listens to `UPDATE` and `INSERT` events on the `agent_runs` table.
2. **State Management**: Maintain a React state array of `AgentRun` objects. Update this array as real-time events arrive.
3. **Progress Calculation**: Derive the overall progress percentage dynamically (e.g., 4 agents done out of 6 total steps = 66%).
