# Phase 14: Real-Time UX & Polish
## Plan 14-02: Streamline Dashboard Agent UI

### Objective
Simplify the Agent Status Grid in the `ActivePipeline.tsx` component so it only shows the relevant agents for the current query, removing clutter and confusion.

### Context & Rationale
The user noted: "6 7 tane isim var orda, mevcut agentlarin adi gozuksun ekstra bir sey gozukmesin" (There are 6-7 names there, only show the existing agents). 
Currently, `ActivePipeline.tsx` hardcodes a grid of 7 agents (`orchestrator`, `search`, `news`, `market`, `macro`, `analyst`, `writer`). For a topic query, the `market` agent is skipped. Seeing "Orkestratör" or "Rapor Üretici" as separate agents might be confusing to an end-user who just wants to know what data is being gathered.

### Scope
- Refactor the hardcoded `AGENT_CONFIG` in `ActivePipeline.tsx`.
- Dynamically render the agent icons based on the actual `agent_runs` records created for that specific session.
- Hide internal infrastructure steps (like `orchestrator` or `writer`) from the grid if they cause clutter, or group them logically into "Veri Toplama" (Data Gathering) and "Sentez" (Synthesis) phases.

### Implementation Steps
1. **Dynamic Rendering**: Map over the unique `agent_type` values present in the real-time `agent_runs` data instead of the static object keys.
2. **Clean Naming**: Ensure the labels are user-friendly (e.g., "KAP Verisi", "Haberler", "Piyasa", "Makro", "Yapay Zeka").
3. **Filter Internal Logs**: If the orchestrator emits a generic log, don't show an "Orchestrator" badge in the UI; just show it as a system message.

### Verification (Goal-Backward)
- [ ] If a user searches for a general topic, does the "Piyasa" (Market) agent cleanly disappear from the grid?
- [ ] Is the number of visible agents reduced to just the core value-drivers (KAP, News, Market, Macro, AI)?
