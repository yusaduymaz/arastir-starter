# Phase 12: Data Reliability & Vercel Compatibility
## Plan 12-04: Session Deletion API & User Data Management

### Objective
Implement an API endpoint and frontend UI to allow users to delete their past research sessions. This fulfills a crucial SaaS requirement for user data management.

### Context & Rationale
Currently, users can create research sessions, but there is no way to delete them from their history. As part of wrapping up the core SaaS features before launch, users must have the ability to manage their data. Deleting a session should cascade and delete the associated `agent_runs` and `agent_logs`.

### Scope
- Create a new API route `DELETE /api/research/[id]`.
- Enforce authentication and authorization (users can only delete their own sessions).
- Update the `HistoryTable` component to include a "Delete" action button.
- Ensure the Supabase foreign key constraints (`ON DELETE CASCADE`) are correctly functioning for `agent_runs` and `agent_logs`.

### Implementation Steps
1. **API Endpoint**: Implement the DELETE route using the Supabase client.
2. **Security Check**: Verify `user_id` matches the currently authenticated Clerk user.
3. **Frontend Action**: Add a trash icon button to the `HistoryTable` rows. Implement an optimistic UI update or a revalidation of the history list upon successful deletion.

### Verification (Goal-Backward)
- [ ] Can a user click "Delete" on a history item and see it removed from the UI?
- [ ] Does the API successfully delete the record from the `research_sessions` table?
- [ ] Are the related `agent_runs` and `agent_logs` automatically deleted from the database?
- [ ] Is a user prevented from deleting another user's session?
