# Phase 12: Data Reliability & Vercel Compatibility
## Plan 12-01: Replace Puppeteer KAP Scraper with HTTP API

### Objective
Remove `puppeteer` and `puppeteer-extra` dependencies from the KAP search agent. Instead, interact directly with the KAP HTTP JSON API (`/api/disclosure/search`) to retrieve company disclosures.

### Context & Rationale
Currently, the KAP agent uses a headless browser (Puppeteer) to navigate `kap.org.tr` and extract DOM elements. This is highly problematic for Vercel deployment, as serverless functions have strict size limits (50MB) and cannot easily bundle Chromium. Furthermore, DOM scraping is slow, fragile, and prone to breaking on minor UI changes. Utilizing the official (or undocumented but stable) HTTP API is faster, lighter, and completely Vercel-compatible.

### Scope
- Analyze the HTTP requests made by `kap.org.tr` when searching for a ticker's disclosures.
- Refactor `src/lib/kap/client.ts` to use `fetch` targeting the JSON endpoints instead of Puppeteer.
- Remove `puppeteer` dependencies from `package.json`.
- Ensure the `KAPDisclosure` type maps correctly to the new JSON response structure.

### Implementation Steps
1. **API Discovery**: Implement a fetch request to the KAP search endpoint with a JSON payload for the specific ticker.
2. **Refactor Client**: Update `fetchKapData` to execute the HTTP request and parse the JSON response.
3. **Clean Up**: Run `npm uninstall puppeteer puppeteer-extra puppeteer-extra-plugin-stealth`.
4. **Agent Update**: Ensure `src/agents/search-agent.ts` handles the new client seamlessly.
