# Plan 20-02: Supabase RPC ile Atomik Token Ledger

## Objective
Token düşüm işlemini API/Server belleğinden çıkararak, Supabase (PostgreSQL) üzerinde bir Stored Procedure (RPC) ile Transaction içinde atomik hale getirmek, böylece eşzamanlı isteklerdeki "Race Condition" hatalarını önlemek.

## Context & Rationale
Şu an `route.ts` içinde önce bakiye okunup (`select balance`), sonra kod içerisinde sıfırdan büyük mü kontrol ediliyor, uygunsa `insert amount = -1` atılıyor. Eşzamanlı (concurrent) 10 istek atılırsa hepsi eski bakiyeyi okur ve hepsi işlemi başlatır; böylece token limiti kolayca aşılabilir. İşlemin tamamen veritabanında, bir Transaction kullanılarak kilit (row-lock) mekanizmasıyla yapılması gerekir.

## Implementation Steps

1. **Supabase SQL Migration Oluşturulması:**
   - Yeni bir migration dosyası hazırlayın. `deduct_token_if_sufficient` isimli bir fonksiyon yaratın.
   - Fonksiyon `p_user_id` (uuid), `p_amount` (int), `p_description` (text) parametreleri almalı.
   - `token_ledger` (veya `user_balances` view'u altındaki ledger toplamları) üzerinden bakiyeyi kontrol etmeli.
   - Bakiye >= `p_amount` ise, `token_ledger`'a negatif işlemi yazmalı ve `true` dönmeli. Değilse `false` dönmeli (veya exception fırlatmalı).

2. **API Rotasında RPC Çağrısına Geçiş:**
   - `src/app/api/research/route.ts` dosyasında `balanceRow` okuma ve kontrol etme mantığını kaldırın.
   - Sadece `supabase.rpc('deduct_token_if_sufficient', { ... })` fonksiyonunu çağırın.
   - Dönen sonuca göre işleme devam edin veya 403 HTTP koduyla "Yetersiz bakiye" yanıtı dönün.

## Execution
- `deduct_token_if_sufficient` RPC'si Supabase'e uygulandı.
- Kod tarafındaki manuel balance check kaldırılıp yerine bu RPC fonksiyonu konuldu.

## Verification
- Test ortamında bir kullanıcıya 1 adet token verin.
- Paralel olarak (Promise.all vb. ile) 5 adet araştırma isteği gönderin.
- Sadece ilk isteğin başarılı olması ve geri kalanların 403 alması, `token_ledger`'da sadece 1 adet negatif kayıt oluşması beklenir.
