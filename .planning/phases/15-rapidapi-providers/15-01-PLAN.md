# Plan: 15-01 - Shared Core Infrastructure, Vitest Setup & Guard Test

**Phase**: 15 - External Market Data APIs (RapidAPI Providers)
**Goal**: Establish the foundational components for the RapidAPI client layer, including shared error handling, caching, rate limiting, and environment variable validation.

## Rationale
Before implementing individual provider clients, we need a robust, shared infrastructure. This ensures consistency, reduces code duplication, and makes the entire system more resilient and easier to maintain. This plan tackles the cross-cutting concerns that will affect all 12 upcoming clients.

## Key Deliverables
1.  **Unified Error Type**: A single `RapidApiError` class to standardize error reporting from all clients.
2.  **Environment Variable Check**: A mechanism to validate the presence of `RAPIDAPI_<PROVIDER>_KEY` environment variables at application startup.
3.  **Core API Client**: A shared `createRapidApiClient` factory function to handle authentication, base URL, and headers.
4.  **Rate Limiter & Cache**: A middleware or wrapper to manage rate limiting and in-memory caching for API requests.
5.  **Vitest Setup**: Configuration of Vitest for running smoke tests against the live APIs, controlled by a feature flag.
6.  **Guard Test**: A single test that verifies the env check guard works correctly by attempting to use a client with a missing API key.

## Implementation Steps

### 1. Create Shared Types and Errors
- **File**: `src/lib/rapidapi/types.ts`
- **Actions**:
    - Define a `RapidApiError` class that extends `Error`. It should accept `provider`, `status`, `code`, and optional `retry_after` properties.
    - Define a `RapidApiClient` interface with `get` and `post` methods.
    - Define a `RapidApiProvider` enum with all 12 provider names.

### 2. Implement Environment Variable Check
- **File**: `src/lib/env-check.ts` (modify existing)
- **Actions**:
    - Create a function `checkRapidApiKeys()` that iterates through the `RapidApiProvider` enum.
    - For each provider, check if `process.env.RAPIDAPI_<PROVIDER>_KEY` exists.
    - If a key is missing, log a warning to the console indicating that the provider will be disabled. Do not throw an error, to allow for graceful degradation.

### 3. Build the Core API Client Factory
- **File**: `src/lib/rapidapi/core.ts`
- **Actions**:
    - Create a factory function `createRapidApiClient(provider: RapidApiProvider, baseUrl: string)`.
    - This function should:
        - Retrieve the API key from `process.env`.
        - If the key is missing, return a "short-circuited" client that immediately returns a `RapidApiError` of kind `missing_key` for any method call.
        - If the key is present, return a `fetch`-based client (using a library like `ky` or `axios` if already in the project, otherwise plain `fetch`).
        - The client should automatically add the `x-rapidapi-host` and `x-rapidapi-key` headers to every request.

### 4. Implement Rate Limiter and Cache
- **File**: `src/lib/rapidapi/middleware.ts`
- **Actions**:
    - Use a library like `p-limit` for rate limiting and `lru-cache` for in-memory caching.
    - Create a function `withMiddleware(client: RapidApiClient, options: { rateLimit?: number, cacheTTL?: number })`.
    - This function will wrap the client's `get` method to:
        1. Check the cache for a response before making a request.
        2. If not cached, acquire a rate-limit token.
        3. Make the API call using the wrapped client.
        4. Store the successful response in the cache.
        5. Release the rate-limit token.

### 5. Configure Vitest for Live Tests
- **File**: `vitest.config.ts` (modify existing)
- **Actions**:
    - Add a new test environment or setup file that only runs tests if `process.env.RAPIDAPI_LIVE_TESTS === '1'`.
    - Ensure environment variables are loaded correctly for the test runs.

- **File**: `src/lib/rapidapi/guard.test.ts`
- **Actions**:
    - Write a test that does **not** set `RAPIDAPI_LIVE_TESTS`.
    - Instantiate a client for a dummy provider (e.g., `CNBC`) without setting its API key.
    - Call a method on the client and assert that it returns a `RapidApiError` with the `kind` set to `missing_key`.

## Success Criteria Verification
- After implementation, run `npm test`. The `guard.test.ts` should pass, confirming the env check and error handling works as expected. The core infrastructure is now ready for the individual provider clients.
