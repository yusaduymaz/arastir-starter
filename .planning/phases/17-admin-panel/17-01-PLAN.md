# Phase 17-01: Admin Panel - Database & Backend Foundation

## Goal
To establish the necessary database schema and backend infrastructure to support the Admin Panel. This includes creating user roles, audit trails, and the API endpoints required for managing users and research sessions.

## Tasks

### Task 1: Apply Database Migrations
- **Description**: Apply the newly created SQL migration files to the Supabase database.
- **Files**:
  - `supabase/migrations/20260517000001_add_user_roles.sql`
  - `supabase/migrations/20260517000002_create_admin_audit_log.sql`
- **Acceptance Criteria**:
  - The `users` table has a `role` column of type `app_role`.
  - The `admin_audit_logs` table is created with the correct schema.
  - RLS policies are in place for both tables.

### Task 2: Implement Role-Based Access Control (RBAC)
- **Description**: Update the existing authentication and API middleware to be role-aware.
- **Files to Modify**:
  - `src/middleware.ts`
  - `src/lib/supabase/utils.ts` (or equivalent auth utility)
- **Acceptance Criteria**:
  - A function `isAdmin()` or similar utility is created to check if the current user has the `admin` role.
  - The middleware protects all `/api/admin/*` routes, ensuring only users with the `admin` role can access them.
  - Non-admin access attempts to admin routes should result in a 403 Forbidden error.

### Task 3: Develop Admin API Endpoints - User Management
- **Description**: Create the backend API routes necessary for admins to perform CRUD operations on users.
- **New Files**:
  - `src/app/api/admin/users/route.ts` (for listing and creating users)
  - `src/app/api/admin/users/[userId]/route.ts` (for getting, updating, and deleting a single user)
- **Acceptance Criteria**:
  - `GET /api/admin/users`: Returns a paginated list of all users.
  - `POST /api/admin/users`: Creates a new user (optional, depending on requirements).
  - `GET /api/admin/users/[userId]`: Returns detailed information for a specific user.
  - `PUT /api/admin/users/[userId]`: Updates a user's details (e.g., their role, subscription status).
  - `DELETE /api/admin/users/[userId]`: Deletes or deactivates a user.
  - All actions must be logged to the `admin_audit_logs` table.

### Task 4: Develop Admin API Endpoints - Research Session Management
- **Description**: Create backend API routes for viewing and managing research sessions.
- **New Files**:
  - `src/app/api/admin/sessions/route.ts` (for listing sessions)
  - `src/app/api/admin/sessions/[sessionId]/route.ts` (for getting and canceling a session)
- **Acceptance Criteria**:
  - `GET /api/admin/sessions`: Returns a list of research sessions, with filtering options (by user, status, etc.).
  - `GET /api/admin/sessions/[sessionId]`: Returns detailed information for a specific session, including its logs.
  - `POST /api/admin/sessions/[sessionId]/cancel`: Cancels a running research session.
  - All management actions (like canceling) are logged to `admin_audit_logs`.

### Task 5: Develop Admin API Endpoints - Dashboard Analytics
- **Description**: Create the backend API route to aggregate and deliver data for the analytics dashboard.
- **New Files**:
  - `src/app/api/admin/analytics/route.ts`
- **Acceptance Criteria**:
  - `GET /api/admin/analytics`: Returns a JSON object containing key metrics:
    - Total user count.
    - Number of active subscriptions.
    - Research sessions count (total, completed, failed).
    - Time-series data for user sign-ups and session creation over the last 30 days.
  - The endpoint should be performant and not cause excessive database load.
