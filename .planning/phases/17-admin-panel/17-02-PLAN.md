# Phase 17-02: Admin Panel - Frontend UI Implementation

## Goal
To build a secure, functional, and user-friendly interface for the Admin Panel that consumes the APIs created in Phase 17-01.

## Tasks

### Task 1: Create Admin Layout and Routing
- **Description**: Set up the basic layout, navigation, and routing structure for the admin section. This section must be access-controlled.
- **New Files**:
  - `src/app/admin/layout.tsx`
  - `src/app/admin/page.tsx` (will redirect to the dashboard)
  - `src/components/admin/AdminSidebar.tsx`
- **Acceptance Criteria**:
  - A new route group `/admin` is created.
  - The `AdminLayout` includes a sidebar for navigation and a main content area.
  - The layout component checks for admin privileges on the client-side and redirects to a 'not authorized' page or the home page if the user is not an admin.
  - The main `/admin` page redirects to `/admin/dashboard`.

### Task 2: Build the Analytics Dashboard Page
- **Description**: Create the main dashboard page that displays key system analytics.
- **New Files**:
  - `src/app/admin/dashboard/page.tsx`
  - `src/components/admin/dashboard/StatCard.tsx`
  - `src/components/admin/dashboard/AnalyticsChart.tsx`
- **Acceptance Criteria**:
  - The page fetches data from the `/api/admin/analytics` endpoint.
  - Key metrics (total users, sessions, etc.) are displayed in `StatCard` components.
  - A chart (`AnalyticsChart`) visualizes user sign-ups and research session trends over time.
  - A loading state is shown while data is being fetched.
  - Errors from the API are handled gracefully.

### Task 3: Build the User Management Page
- **Description**: Create the UI for listing, viewing, and editing users.
- **New Files**:
  - `src/app/admin/users/page.tsx`
  - `src/components/admin/users/UsersDataTable.tsx`
  - `src/components/admin/users/EditUserDialog.tsx`
- **Acceptance Criteria**:
  - The page displays a table of users fetched from `/api/admin/users`.
  - The `UsersDataTable` component should support sorting, filtering by email/name, and pagination.
  - Clicking an 'Edit' button on a user row opens an `EditUserDialog`.
  - Inside the dialog, an admin can change a user's role.
  - All interactions with the API should handle loading and error states.
  - Actions are confirmed with the user (e.g., "Are you sure you want to change this user's role to Admin?").

### Task 4: Build the Research Session Management Page
- **Description**: Create the UI for monitoring and managing research sessions.
- **New Files**:
  - `src/app/admin/sessions/page.tsx`
  - `src/components/admin/sessions/SessionsDataTable.tsx`
- **Acceptance Criteria**:
  - The page displays a table of research sessions from `/api/admin/sessions`.
  - The table should be updated in real-time if possible (using Supabase Realtime or polling).
  - The `SessionsDataTable` supports filtering by status (e.g., running, completed, failed) and searching by user.
  - An admin can click a 'Cancel' button on a running session, which triggers a call to the cancel API endpoint.
  - A 'View Details' button allows the admin to see the full logs or outputs of a session (in a dialog or a separate page).

### Task 5: UI Polishing and Component Library Integration
- **Description**: Ensure the new admin panel is consistent with the existing application's design system and is easy to use.
- **Files to Modify**:
  - All new components in `src/components/admin/`
  - `src/app/globals.css` or Tailwind config if necessary.
- **Acceptance Criteria**:
  - All new components (tables, dialogs, charts) are built using the project's existing UI library (e.g., Shadcn/UI, Material-UI).
  - The UI is responsive and works well on different screen sizes.
  - Consistent styling, spacing, and typography are used throughout the admin panel.
  - Forms include client-side validation where appropriate.
