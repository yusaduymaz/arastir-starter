# Phase 11: Multi-Agent SaaS Architecture Redesign
## Plan 11-02: Core SaaS Services (Ledger Refactoring)

### Objective
Refactor the backend authentication and SaaS services to utilize the new `token_ledger` and `user_balances` view instead of mutating the `tokens_balance` column directly on the `users` table.

### Context & Rationale
With the new DB schema in place (from 11-01), the application code needs to be updated to read from and write to the ledger. When a user signs up via Clerk, they should receive an initial "grant" in the ledger. When a user requests a research report, the system should append a "usage" record (-1 amount) to the ledger rather than doing an `UPDATE users SET tokens_balance = tokens_balance - 1`.

### Scope
- Refactor Clerk Webhook handler to insert initial tokens into `token_ledger`.
- Refactor the Research API (`src/app/api/research/route.ts` or similar) to check the `user_balances` view before starting a job.
- Deduct tokens by inserting a negative `amount` into the `token_ledger`.
- Update the frontend Dashboard header to fetch the balance from `user_balances` instead of the `users` table.

### Implementation Steps

#### 1. Clerk Webhook Refactoring
Locate the Clerk Webhook API route (likely `src/app/api/webhooks/clerk/route.ts`).
- When a user is created (`user.created`), insert them into the `users` table.
- Then, perform a second insert into `token_ledger`: `{ user_id: id, amount: 5, transaction_type: 'grant', description: 'Signup bonus' }`.

#### 2. Token Deduction Logic
In the research initiation endpoint:
- Query the `user_balances` view for the current `user_id`.
- If `balance <= 0`, return `402 Payment Required`.
- If sufficient, insert into `token_ledger`: `{ user_id: id, amount: -1, transaction_type: 'usage', description: 'Research query for ' + ticker }`.

#### 3. Frontend Balance Display
Update the `Dashboard` or `Header` component to fetch the computed balance. This might involve updating a `get-user` server action or API route to `SELECT * FROM user_balances WHERE user_id = ?`.

### Verification (Goal-Backward)
- [ ] When a new user signs up, do they get a 5 token grant in the ledger?
- [ ] When a user runs a report, is a usage record created?
- [ ] Does the UI correctly display the sum of the ledger?
- [ ] Is a user blocked from starting a job if their computed balance is 0?
