# Phase 11: Multi-Agent SaaS Architecture Redesign
## Plan 11-03: Agent Orchestrator Refactor (Run-based Execution)

### Objective
Rewrite the central `orchestrator` logic to interact with the new `agent_runs` table. Agents will no longer update monolithic columns on the `research_sessions` table; instead, they will execute by creating, updating, and completing their specific `agent_runs` records.

### Context & Rationale
In Phase 10, the orchestrator was refactored to use in-process async functions, but it still writes all outputs directly to `kap_data`, `news_data`, etc. on the main `research_history` row. This makes adding new data sources hard and makes it difficult to track exactly when and why a specific agent failed. Moving to an `agent_runs` execution model turns the orchestrator into a true state machine.

### Scope
- Refactor the main `executeResearchPipeline` function (likely in an API route or server action).
- Refactor individual agent wrapper functions (e.g., `src/agents/search-agent.ts`, `news-agent.ts`) to accept a `session_id` and internally manage their `agent_runs` lifecycle (create run -> running -> complete/fail).
- Ensure the Analyst agent fetches its inputs by querying completed `agent_runs` for the given `session_id`, rather than receiving a massive JSON object passed through the stack.
- Emit fine-grained logs to `agent_logs`.

### Implementation Steps

#### 1. Orchestrator Pipeline Refactor
Update the main entry point to:
1. Create a `research_sessions` record.
2. Initialize `agent_runs` records for each required step (e.g., `kap`, `news`, `market`, `macro`).
3. Call the agent functions in parallel or sequence, passing `session_id` and `run_id`.

#### 2. Agent Wrapper Refactor
Modify each agent (e.g., `kapAgent`):
```typescript
async function runKapAgent(sessionId: string, runId: string, ticker: string) {
  // Update status to 'running'
  await supabase.from('agent_runs').update({ status: 'running', started_at: new Date() }).eq('id', runId);
  
  try {
    const data = await fetchKapData(ticker);
    // Update status to 'completed' and save output
    await supabase.from('agent_runs').update({ 
      status: 'completed', 
      output_data: data,
      completed_at: new Date()
    }).eq('id', runId);
  } catch (err) {
    // Update status to 'failed'
    await supabase.from('agent_runs').update({ status: 'failed', error_message: err.message }).eq('id', runId);
    throw err;
  }
}
```

#### 3. Analyst Synthesis Refactor
Update the Analyst agent to pull data from `agent_runs`:
```typescript
const { data: runs } = await supabase.from('agent_runs')
  .select('agent_name, output_data')
  .eq('session_id', sessionId)
  .eq('status', 'completed');
// Map the runs into a synthesis payload
```

### Verification (Goal-Backward)
- [ ] Do agents successfully execute and store their data in `agent_runs.output_data`?
- [ ] Does the orchestrator successfully manage parallel execution and wait for all `agent_runs` to complete?
- [ ] Does the Analyst agent correctly gather data from the database rather than memory?
