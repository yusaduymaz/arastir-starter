# Phase 11: Multi-Agent SaaS Architecture Redesign
## Plan 11-01: Database Architecture Overhaul

### Objective
Completely redesign the Supabase database schema to support an open-closed multi-agent system and robust SaaS billing. Move away from the monolithic `research_history` table toward a highly normalized, event-driven task execution model.

### Context & Rationale
Currently, the `research_history` table contains agent-specific columns (`kap_data`, `news_data`, `market_data`, `macro_data`). This tightly couples the database schema to the specific agents we have today. If we want to add a new agent (e.g., Social Sentiment Agent), we must add a new column. Furthermore, tracking progress with a single `progress` integer and `current_step` string is insufficient for a complex multi-agent system running tasks in parallel or sequence. We need an `agent_runs` table.

For billing, relying solely on a `tokens_balance` integer in the `users` table is prone to race conditions and offers no audit trail. A `token_ledger` table is the industry standard for SaaS token management.

### Scope
- Create a new migration file: `20260516000000_multi_agent_saas_redesign.sql`
- Re-create the core SaaS structures (`users`, `token_ledger`, and a view `user_balances`).
- Create `research_sessions` (replaces `research_history` but without the data columns).
- Create `agent_runs` to track individual agent executions within a session.
- Create `agent_logs` for fine-grained logging per run.
- Migrate existing data from `research_history` to `research_sessions` and `agent_runs` (optional but recommended if data needs preservation, or just start fresh if in alpha).
- Update the TypeScript types in `src/types/research.ts` and `src/types/saas.ts` to match the new schema.

### Implementation Steps

#### 1. SQL Schema Definition
Create the migration `supabase/migrations/20260516000000_multi_agent_saas_redesign.sql` with the following structure:
```sql
-- 1. SaaS Token Ledger
CREATE TABLE token_ledger (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    amount INTEGER NOT NULL, -- positive for top-up/grant, negative for usage
    transaction_type TEXT NOT NULL CHECK (transaction_type IN ('grant', 'purchase', 'usage', 'refund')),
    description TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::TEXT, NOW()) NOT NULL
);

CREATE OR REPLACE VIEW user_balances AS
SELECT user_id, SUM(amount) as balance
FROM token_ledger
GROUP BY user_id;

-- 2. Multi-Agent Orchestration Schema
CREATE TABLE research_sessions (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    user_id TEXT NOT NULL,
    query TEXT NOT NULL,
    extracted_ticker TEXT,
    status TEXT DEFAULT 'pending' NOT NULL CHECK (status IN ('pending', 'running', 'completed', 'failed')),
    progress INTEGER DEFAULT 0,
    result_url TEXT,
    error_message TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::TEXT, NOW()) NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::TEXT, NOW()) NOT NULL
);

CREATE TABLE agent_runs (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    session_id UUID NOT NULL REFERENCES research_sessions(id) ON DELETE CASCADE,
    agent_name TEXT NOT NULL, -- e.g., 'kap', 'news', 'market', 'macro', 'analyst', 'writer'
    status TEXT DEFAULT 'pending' NOT NULL CHECK (status IN ('pending', 'running', 'completed', 'failed', 'skipped')),
    input_data JSONB,
    output_data JSONB,
    error_message TEXT,
    started_at TIMESTAMP WITH TIME ZONE,
    completed_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::TEXT, NOW()) NOT NULL
);

CREATE TABLE agent_logs (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    run_id UUID NOT NULL REFERENCES agent_runs(id) ON DELETE CASCADE,
    log_level TEXT DEFAULT 'info',
    message TEXT NOT NULL,
    metadata JSONB,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::TEXT, NOW()) NOT NULL
);
```
