# Phase 11: Multi-Agent SaaS Architecture Redesign
## Plan 11-04: Frontend Dashboard Refactor (Realtime Subscriptions)

### Objective
Update the frontend dashboard to consume the new `research_sessions` and `agent_runs` tables using Supabase Realtime. This will provide users with a professional, step-by-step loading state UI instead of a basic 1-100% progress bar.

### Context & Rationale
Currently, the frontend listens to a single `progress` integer and `current_step` string. With the new schema, the frontend can subscribe to the `agent_runs` table to see exactly which agents are currently running, which have completed, and display any specific agent logs in a terminal-like UI or beautiful checklist. This drastically improves the UX by making the AI feel "alive".

### Scope
- Refactor the `Dashboard` and `History` views to query `research_sessions`.
- Create a new `ActiveSessionView` component that subscribes to `agent_runs` changes via Supabase Realtime.
- Map the agent states to a UI (e.g., Checkmarks, Spinners, Error icons).
- Show realtime terminal logs by subscribing to `agent_logs`.

### Implementation Steps

#### 1. Session List View
Update the main dashboard to load user queries from `research_sessions`. The table columns will be mostly the same, but the underlying data type `ResearchSession` will no longer have massive data columns attached.

#### 2. Realtime Progress Component
Create a `PipelineStatus` component that takes a `sessionId`.
- Initialize a Supabase channel: `supabase.channel('public:agent_runs').on('postgres_changes', { event: '*', schema: 'public', table: 'agent_runs', filter: 'session_id=eq.'+sessionId }, handlePayload).subscribe()`.
- Render a list of all known agents for the session:
  - `[x] KAP Agent (Completed)`
  - `[spinner] News Agent (Running)`
  - `[ ] Analyst Agent (Pending)`

#### 3. Realtime Logs Component
Optionally, create an `AgentLogs` terminal component that subscribes to the `agent_logs` table for the active `session_id`, streaming the thoughts/actions of the AI in real-time.

### Verification (Goal-Backward)
- [ ] Can the user see a list of their past `research_sessions`?
- [ ] When a new session is started, does the UI show the parallel progress of the different `agent_runs`?
- [ ] Do updates appear instantly via WebSockets without page reload?
- [ ] If an agent fails, does it show an error state for that specific agent while the overall session handles the error gracefully?
